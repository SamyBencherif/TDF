TDF Language and Development Manual

Introduction

This document contains comprehensive instructions regarding use, development, and implementation of the Test Descriptor Format. This document is required for anyone tasked with developing test and abort sequences, proving TDF code, making use of TDF libraries, or developing mission critical peripheral technologies. Rule of thumb: if your project description includes TDF, this document is for you.

Naming Convention

First a number of points.

  1. Use snake casing.
  2. Do NOT use abbrevations for any significant constructs. This includes function names, structure fields, and global variables.
  3. Abbreviations may be used sparingly for temporary pointers, such as function arguments where typename is sufficiently descriptive.
  4. Do NOT use ambiguous abbreviations. For example "dev" can be short for either "development" or "device".
  5. Naming convention does NOT replace documentation and comments.

TDF programs are stored in memory and processed in a manner that is reminiscent of Object-Orientation. This design choice reduces the amount of code that handles bytecode-like objects. Object- Orientation structures may or may not be involved in program execution. Use of the structures in analysis, profiling, and compilation, however is uncontested.

Development

This manual contains information about development that will remain constant throughout the project's lifetime. Standards, practices, will be included here as well. It is important to read this section completely before moving on to the next reference document. Project structure, compilation, development tools, and integration will be discussed. Once you have completed this section, see development_status.txt for information about the current steps and history of development; most recent comments are stored at the top of the file.

The project root may only contain two files. The README.md and the Makefile. The unusual casing of both these files is intentional.
